- **Пункт 2 — `src/stomengine/lowmem.py` (НОВЫЙ).** Корень OOM — буфер логитов 49×533×533×467 ≈ **13 ГБ**
  (сам по себе > 8 ГБ), а дефолтный экспорт nnU-Net апкастит его в float32 (~26 ГБ). Чистый Z-бандинг НЕ
  спасает: halo (2×112 срезов) сам превышает бюджет, т.к. плоскость XY полная (27.8 МБ/срез). Решение —
  **двойное:** (1) буфер sliding-window — `numpy.memmap` (ОС вытесняет холодные Z-слэбы на диск, в RAM
  только рабочий набор + n_predictions ~0.3 ГБ); (2) **chunked argmax по Z** на сетке сети → uint8 метки
  (никогда float32), затем **nearest-ресемпл МЕТОК** (1 канал, не 49) на геометрию входа через
  `insert_crop_into_image` + `transpose_backward` (та же бухгалтерия nnU-Net, но на uint8).
  `MemmapLogitsPredictor` = копия `_internal_predict_sliding_window_return_logits` из **nnunetv2 2.8.0**
  (запинено; при апгрейде nnU-Net — пере-сдиффить) с одной правкой: аккумулятор memmap-backed.
  Тумблер `STOM_DISABLE_LOWMEM=1` → in-RAM путь. Отклонение от nnU-Net: argmax-then-nearest вместо
  resample-then-argmax (на апсемпле 0.3→0.2мм — субвоксельный джиттер на границах, неважно).
- **Пункт 3 — `ToothFairy2Runner` (`runner.py`) + `tf2_labels.py` (НОВЫЙ, 42 метки = 10 анатомия + 32 зуба
  FDI; NA-зазоры 19/20/29/30/39/40 опущены).** Отличия от DentalSegmentator: ZScore per-image → **БЕЗ
  гармонизации HU**, только clamp out-of-FOV паддинга (`<-1000`→`-1000`, `clamp_air_padding`);
  `checkpoint_best.pth`, folds=(0,); экспорт через lowmem. Выбор модели в `cli.py`: `STOM_MODEL=toothfairy2`
  (или `tf2`) → TF2-runner + TF2-метки + путь `models/Dataset112_ToothFairy2/...`. `InProcessEngine(labels=)`
  теперь параметризован картой меток.
- **Валидация:** `tf2_data/validate_runner.py` (gitignored) — синтетический мини-CBCT через ОБА пути на
  реальном ResEnc-L: shape==вход, геометрия сохранена, прогресс ок, **memmap и in-RAM дают идентичные
  метки**, без OOM. Тесты: **240 passed** (+21: `test_lowmem`, `test_tf2_labels`, `test_tf2_cli_wiring`,

### ⚠️ OOM-гоча и фикс (важно для продукта)
Экспорт 49-классовых логитов на полном разрешении = **26–39 ГБ** → 3 прогона упали по OOM (SIGKILL без
трейсбека). Корень: `resampling_fn_probabilities` апкастит float16-логиты (~13 ГБ) в float32 (~26 ГБ),
держа оба. **Фикс** (`tf2_data/run_full_v2.py`): подать вход уже на 0.3 мм (= спейсинг сети) → целевая
форма ресемпла совпадает → ресемпл = тождество, пропускаем его (патчим property на классе
`ConfigurationManager`, т.к. сеттера нет). Пик упал до ~14 ГБ. **НО: целевые ПК юзеров ~6–8 ГБ RAM →
ToothFairy2 как есть на них НЕ запустится** → нужен экспорт чанками по Z (argmax по срезам).

## ✅ v0.1.9 — фикс зависшего окна «Локальный движок (Не отвечает)» после установки
Причина (по скрину юзера): движок скачивался и распаковывался **успешно**, но финальное окно
«Движок установлен» появлялось пустым белым и Windows вешал на него «(Не отвечает)». Юзер
описал это как «локальный движок не отвечает» — но такой строки в коде нет; это была штатная
подпись Windows к зависшему окну.

Корень — гонка потоков в Qt (`main_window.py`): `worker.done` был подключён через **context-less
`lambda`** (`lambda eng: self._on_install_done(eng, updated=clean)`). Сигнал, подключённый к лямбде
без объекта-контекста, в Qt выполняется **в потоке, испустившем сигнал** — т.е. в рабочем потоке
установки. `_on_install_done` оттуда создавал `QMessageBox` (виджет Qt) в НЕ-GUI-потоке → окно без
своего цикла событий → пустое + «(Не отвечает)». Всплыло только сейчас, потому что в прошлых
сборках движок не докачивался и ветка `done` впервые отработала. Путь `failed` («Не удалось
установить») был исправен — он подключён к методу `self` (маршалится в UI-поток).

Сделано:
- **main_window:** `worker.done.connect(self._on_install_done)` (bound-метод → AutoConnection →
  очередь в UI-поток). Флаг install-vs-update теперь хранится в `self._installing_clean`
  (выставляется в `_start_engine_install`), а `_on_install_done(updated=None)` берёт его как фолбэк.
- **main_window:** заодно `thread.finished` (старт-чеки) переведён с context-less лямбды на
  bound-метод `_on_check_thread_finished` (через `self.sender()`) — та же категория бага, чистка
  списка потоков теперь в UI-потоке.
- **Тесты:** +1 регрессионный (`test_install_done_uses_stored_clean_flag`) — сигнал воркера зовёт
  `_on_install_done(engine)` без `updated`, обработчик должен взять режим из `_installing_clean`
  и показать сообщение обновления, а не установки. Полный прогон **188 passed**.
- **bump** `0.1.8 → 0.1.9` (`pyproject.toml` + `packaging/stom-client.iss`). Баг чисто клиентский,
  но релиз по тегу пересобирает ОБЕ сборки (.exe + engine-pack) и публикует манифест v0.1.9.

## ✅ v0.1.7 — живой прогресс сегментации в процентах (выпущено, smoke зелёный)
Причина: при локальной сегментации не было обратной связи во время долгого CPU-инференса —
было непонятно, началась она вообще или нет. Теперь GUI показывает «Сегментация… NN%»,
процент растёт по мере прохождения тайлов nnU-Net (а до первого тайла — «Сегментация… (идёт расчёт)»).

Сделано (коммит `6f53d66`, выпущено в v0.1.7):
- **runner:** `DentalSegmentatorRunner.predict(progress=)` направляет tile-loop nnU-Net в колбэк
  `(done, total)`, подменяя символ `tqdm` в модуле инференса на крошечный счётчик-шим (без форка
  и без копирования версионных внутренностей). `FakeRunner` шлёт один шаг — чтобы проводка
  проверялась end-to-end.
- **engine:** `SubprocessEngine` теперь **стримит** stdout engine-pack в потоке-читателе (Popen,
  stderr слит в stdout, чтобы не было дедлока на полном пайпе) и парсит строки `PROGRESS d t`
  вживую — вместо `subprocess.run(capture_output)`, который блокировал до выхода процесса.
  Непрогрессный вывод по-прежнему копится для диагностики ошибок. `InProcessEngine` пробрасывает
  progress, когда задан.
- **cli:** `stom-engine predict` печатает flush-нутые строки `PROGRESS d t` по тайлам.
- **app_controller/main_window:** `submit(progress=)` → Qt-сигнал `_SubmitWorker` → статус-лейбл,
  маршалится в UI-поток.
- **Важно:** прогресс задействует ОБЕ половины — стриминг-ридер на стороне клиента (.exe) и эмиттер
  `PROGRESS` на стороне engine-pack. Поэтому v0.1.7 пересобирала и публиковала обе сборки.
