# Дизайн: Аналог DiagnoCAT для сегментации CBCT (MVP)

**Дата:** 2026-06-09
**Статус:** черновик дизайна (одобрен по секциям, ожидает финального ревью)
**Домен:** стоматология / челюстно-лицевая CBCT

## 1. Цель и контекст

Коммерческий продукт для автоматической AI-сегментации стоматологических
CBCT-исследований — аналог DiagnoCAT. Гибридная архитектура: десктоп-клиент для
просмотра и правки + облако для тяжёлой AI-обработки.

**Основной режим:** полностью автоматическая AI-сегментация (зубы по FDI,
нижнечелюстной канал, челюсти). Ручная правка масок входит в первую версию.

**Базовый ML-подход:** старт с open-source предобученных моделей (nnU-Net на
датасете ToothFairy2), дообучение по мере появления собственных данных. Своей
ML-команды и размеченных датасетов пока нет.

## 2. Границы MVP

**Входит:**
- Десктоп-клиент: импорт DICOM-серии CBCT, просмотр 2D-срезов
  (axial/sagittal/coronal) и 3D-объёма, отправка в облако, наложение масок,
  ручная правка масок, измерения, экспорт.
- Облачный сервис: приём исследования, AI-сегментация (nnU-Net/ToothFairy2),
  возврат масок.
- Хранилище: объектное (S3-совместимое) для томов/масок + БД метаданных.

**НЕ входит (отложено, но архитектурно заложено):** автодетекция патологий,
генерация текстовых заключений, многопользовательские роли, интеграция с PACS,
мобильный доступ, медицинская сертификация.

**Поток (одно предложение):** врач открывает DICOM в клиенте → жмёт
«Сегментировать» → клиент заливает том в облако → облако прогоняет nnU-Net →
клиент скачивает маски → врач смотрит, правит кистью, измеряет, экспортирует.

## 3. Архитектура верхнего уровня

Три независимых модуля, общающихся через явные интерфейсы:

| Слой | Технология | Роль |
|---|---|---|
| Десктоп-клиент | Python + PySide6 + VTK + ITK | UI, локальная обработка, правка |
| API-сервер | Python + FastAPI | Приём задач, хранение, статусы (не считает) |
| Воркер сегментации | Python + PyTorch + nnU-Net (GPU) | AI-сегментация (stateless) |
| Очередь | Redis + RQ/Celery | Развязка API и GPU-воркера |
| Объектное хранилище | S3 / MinIO | Тома и маски |
| БД | PostgreSQL | Пациенты, исследования, задачи, метаданные |

Клиент общается с облаком только через REST/HTTPS API. API оркестрирует, но сам
не считает. Воркер берёт задачи из очереди и кладёт результат в хранилище.

**Обоснование стека клиента (Python + PySide6 + VTK/ITK):** единый язык с
облаком (общие модели данных, парсинг, формат масок); зрелая медицинская
экосистема (VTK — объёмный рендеринг и срезы, ITK — DICOM/ресемплинг); полный
контроль над брендированным UX (в отличие от модуля внутри 3D Slicer); узкие
места выносимы в C++ позже. Electron/web отклонён из-за слабого объёмного
рендеринга и менее зрелого DICOM-тулинга в JS.

## 4. Десктоп-клиент (подсистемы)

**Слой данных:**
- `DicomLoader` — читает DICOM-серию через ITK, собирает 3D-том, валидирует
  CBCT и целостность серии. Вход: путь. Выход: `Volume`.
- `Volume` — иммутабельный контейнер вокселей + геометрия (spacing, origin,
  direction). Единый источник правды о пространстве; маски обязаны совпадать.
- `SegmentationMask` — мульти-лейбл маска той же геометрии + карта
  `label_id → {название FDI/структура, цвет, видимость}`.

**Слой визуализации (VTK):**
- `SliceView` — один 2D-вид: срез + полупрозрачные маски, скролл по срезам,
  window/level, синхронизированный курсор-крест.
- `VolumeView` — 3D объёмный рендеринг + поверхности масок (marching cubes).
- `ViewerLayout` — «крест» 2×2: три 2D-вида + 3D, синхронизация курсора.

**Слой взаимодействия:**
- `CloudClient` — единственная точка общения с облаком: `upload_study()`,
  `start_segmentation()`, `poll_status()`, `download_masks()`. Прячет HTTP.
- `MaskEditor` — кисть/ластик по активному лейблу, 2D-порог в области,
  undo/redo через стек операций. Изменения сразу видны во всех видах.
- `MeasurementTool` — линейные/угловые измерения в мм (с учётом spacing).

**Оркестрация:**
- `AppController` — связывает UI-действия с подсистемами, держит состояние
  сессии (текущее исследование, статус облачной задачи).

Принцип границ: подсистемы общаются через `Volume`/`SegmentationMask` и явные
методы, не лезут друг другу внутрь → независимое тестирование.

## 5. Облачный backend

**API-сервер (FastAPI):**
- `POST /studies` — приём тома (NIfTI), запись в БД, файл в хранилище →
  `study_id`.
- `POST /studies/{id}/segment` — постановка задачи в очередь → `job_id`.
- `GET /jobs/{id}` — статус: `queued / running / done / failed`.
- `GET /studies/{id}/masks` — отдача готовых масок.
- Токен-аутентификация (медданные — обязательно даже в MVP).

**Очередь (Redis + RQ/Celery):** разводит быстрый API и медленный GPU-воркер;
позволяет масштабировать воркеры независимо.

**Воркер сегментации (GPU, stateless):**
задача → скачать том → ресемпл под вход модели → nnU-Net (ToothFairy2) →
постобработка (разделение инстансов зубов, нумерация FDI, сглаживание) →
ресемпл маски обратно в геометрию исходного тома → маска в хранилище → статус
`done`. Падение воркера = задача переедет на другой.

**Хранилище:**
- Объектное (S3/MinIO): тома и маски.
- PostgreSQL: пациенты, исследования, задачи, статусы, ссылки на объекты. Без
  вокселей в БД.

**Формат обмена:** `NIfTI .nii.gz` (компактнее DICOM, одна геометрия, стандарт
ML). Клиент конвертирует DICOM→NIfTI перед загрузкой; маски возвращаются тоже
`.nii.gz`.

**Геометрическая инвариантность (ключевой инвариант):** возвращённая маска
обязана идеально совпадать по spacing/origin/direction с исходным томом.
Воркер гарантирует ресемплингом обратно в исходную геометрию.

## 6. Обработка ошибок

- **Импорт DICOM:** битая/неполная серия, не CBCT, несовместимые срезы →
  `DicomLoader` бросает понятную ошибку, клиент показывает причину, не падает.
- **Сеть/облако:** возобновляемая chunked-загрузка + ретраи в `CloudClient`;
  при потере связи задача в облаке продолжается, клиент переподключается по
  `job_id`.
- **Сегментация упала** (OOM, кривой вход) → воркер ставит `failed` + причину,
  клиент предлагает повтор или ручную сегментацию; очередь не падает.
- **Геометрическое рассогласование** → клиент проверяет совпадение при загрузке
  маски и отвергает несовместимую.

## 7. Безопасность медданных

- HTTPS везде, токен-аутентификация на API.
- Шифрование at-rest в хранилище; опция анонимизации DICOM (вырезание ФИО/ID из
  тегов) на клиенте перед отправкой.
- Изоляция данных по аккаунту/клинике на уровне БД.
- Архитектурный задел под сертификацию (журнал доступа); полноценный аудит-лог
  вне MVP.

## 8. Тестирование

- **Юнит:** `DicomLoader` (тестовые серии), `MaskEditor` (детерминированные
  операции/undo), геометрия `Volume` (round-trip ресемплинг).
- **Контрактные:** `CloudClient` против мок-API (фиксируем формат).
- **Воркер:** прогон на эталонном CBCT → сравнение с верной маской (Dice выше
  порога).
- **Интеграционный:** сквозной DICOM → облако (локальный MinIO+Redis) → маска →
  наложение.
- **Стек:** `pytest`, моки для VTK-рендеринга (тестируем логику, не пиксели).
